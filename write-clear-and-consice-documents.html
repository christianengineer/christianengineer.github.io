How to make your engineering documents more clear and concise

The importance of clear and concise communication grows as you become more senior in your engineering career. It helps your readers understand your thoughts, technical solutions, and decisions quickly. These are the patterns I use in my daily writing to enable faster decision-making amongst other leaders and colleagues.

Convert characters (X) to nouns and actions (Y) to verbs:

  1. X because Y.
  2. Since X, Y.
  3. If X, then Y.
  4. Although X, Y.
  5. X and/but/so Y.

Using these patterns will help break up your writing, engaging your readers with easier to digest sentences, keeping them interested in your message.

They will also make your documentation easier to understand and convey meaning effectively, reducing confusion and misinterpretation.

Your new writing style will have a greater impact on your readers, as they draw attention to specific points and emphasize their importance.

X because Y:

  • "The Next.js page used getInitialProps because it needed to fetch data before it was rendered."

  • "The Next.js app used the built-in CSS-in-JS library because it needed to handle styling dynamically."

  • "The React component re-rendered because the state changed."

Since X, Y:

  • "Since the TypeScript code was strongly typed, the compiler caught type errors during development."

  • "Since the React component used a context hook, it shared state with other components without passing props."

  • "Since the GraphQL API used types, the data was validated before it reached the backend."

  • "Since Next.js used automatic code splitting, the users only downloaded the code they needed."

If X, then Y:

  • "If the React component received new props, then it re-rendered with updated data."

  • "If the TypeScript code had type errors, then the compiler wouldn't generate JavaScript code."

  • "If the Next.js app used client-side rendering, then the pages would update without reloading the page."

Although X, Y:

  • "Although the browser faced compatibility issues, the JavaScript code was written to support multiple browsers."

  • "Although the TypeScript code was more complex, it had better type safety and prevented runtime errors."

  • "Although the GraphQL API had strict requirements, it was easier to scale and maintain."

X and/but/so Y:

  • "The GraphQL API used subscriptions and real-time updates, and the frontend was always up to date."

  • "The Next.js pages used the shared components, but they were also customizable for each specific page."

  • "The React component used a state hook and re-rendered when the state changed, but it also used React.memo to optimize for performance."

  • "The React component used a context hook and shared state with other components, so the data was consistent throughout the application."

  • "The TypeScript code was written with the latest syntax and features, and the compiler compiled it to JavaScript that was compatible with older browsers."

  • "The GraphQL API used types and strict validation, and the data was always consistent and reliable."

By using the patterns above, you can effectively communicate technical concepts and solutions, and leave a lasting impression on your readers. Rewrite your sentences to include characters as nouns and actions as verbs. This will improve the clarity and conciseness of all your engineering documents.
